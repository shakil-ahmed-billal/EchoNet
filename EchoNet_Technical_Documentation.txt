1. Project Overview


EchoNet is a full-stack social media platform designed to deliver a rich, real-time social experience across web and mobile browsers. It supports three distinct user roles — Admin, Moderator, and User — each with carefully scoped permissions to ensure safe, scalable community interaction.

1.1 Core Objectives
	•	Provide a seamless platform for creating posts, comments, likes, and replies
	•	Enable real-time text chat, video calls, and audio calls between users
	•	Enforce role-based content moderation and user management
	•	Deliver instant push notifications across devices
	•	Ensure a mobile-optimised, SEO-friendly responsive design
	•	Support optional in-app payments via SSLCommerz

5. Project Workflow


5.1 User Registration & Authentication Flow
	•	User submits email/password or clicks OAuth provider (Google/Facebook)
	•	Better-Auth validates credentials, creates session, issues JWT access token + refresh token
	•	Access token stored in httpOnly cookie; refresh token persisted in DB via Prisma
	•	Subsequent API requests include access token; middleware validates and attaches user context
	•	On token expiry, silent refresh endpoint issues new access token using valid refresh token

5.2 Post Lifecycle
	•	User composes post (text + optional media) on the feed page
	•	Media files upload to cloud storage (e.g. AWS S3 / Cloudinary); URLs returned
	•	POST /api/posts creates Post record in PostgreSQL via Prisma
	•	Socket.IO emits post:created event to connected followers' feeds
	•	Admin/Moderator can change post status or delete via dashboard

5.3 Real-time Chat Flow
	•	User opens chat with another user; client joins Socket.IO room (user:{id})
	•	On message send, socket emits message:send with content and receiverId
	•	Server saves Message to PostgreSQL, updates Redis cache for recent messages
	•	Server emits message:receive to receiver's socket room
	•	Unread badge count updated via notification socket event

5.4 Video/Audio Call Flow
	•	Caller clicks call button; client sends call:initiate event with offer SDP via Socket.IO
	•	Server relays offer to callee's socket room
	•	Callee accepts; client sends call:answer with answer SDP
	•	ICE candidates exchanged via call:ice-candidate events through the signalling server
	•	WebRTC peer connection established; media streams bypass server (P2P)
	•	Either party ends call; call:end event cleans up state on both sides


6. Integration Details


6.1 OAuth Integration (Better-Auth)
Better-Auth handles the OAuth 2.0 flow for Google and Facebook. Configure provider credentials in environment variables. Better-Auth creates or updates the User record on first sign-in and generates a JWT session.

Provider
ENV Variable
Scope Required
Google
GOOGLE_CLIENT_ID / GOOGLE_CLIENT_SECRET
openid email profile
Facebook
FACEBOOK_APP_ID / FACEBOOK_APP_SECRET
email public_profile

6.2 Push Notifications (Firebase Cloud Messaging)
	•	On login, client requests FCM permission and obtains a device registration token
	•	Token stored against User record in PostgreSQL
	•	Server-side notification.service.ts uses Firebase Admin SDK to send targeted push messages
	•	Events triggering push: new like, comment, message, announcement, incoming call

6.3 Media Upload
	•	Frontend uses presigned URLs (from cloud storage service) or multipart form upload
	•	Supported MIME types: image/jpeg, image/png, image/webp, video/mp4
	•	File size limits recommended: Images 5MB, Videos 100MB
	•	Returned CDN URLs stored in Post.mediaUrls array in PostgreSQL

6.4 Payment Integration — SSLCommerz (Optional)
	•	Integrate SSLCommerz SDK for subscription or donation flows
	•	Create payment session on backend, redirect user to SSLCommerz hosted page
	•	Handle IPN (Instant Payment Notification) webhook to confirm and fulfil payments
	•	Supported methods: Credit/Debit card, Mobile banking (bKash, Nagad), Net banking

8.2 WebRTC Configuration
	•	STUN servers: Use Google's public STUN (stun.l.google.com:19302) for development
	•	TURN servers: Required for production environments behind symmetric NAT — use Coturn or a managed TURN service
	•	ICE candidates are relayed through Socket.IO signalling server (not peer-to-peer)
	•	Media streams (audio/video) travel directly peer-to-peer after connection establishment

10. Error Handling & Debugging Strategies


10.1 Backend Error Handling
	•	Global Express error-handling middleware catches all unhandled errors and returns structured JSON responses
	•	HTTP status codes follow REST conventions: 400 Bad Request, 401 Unauthorised, 403 Forbidden, 404 Not Found, 500 Internal Server Error
	•	Prisma errors mapped to user-friendly messages: P2002 (unique constraint) → 409 Conflict
	•	Async route wrappers (asyncHandler) ensure uncaught promise rejections are forwarded to error middleware

Standard error response shape:

Type
Description

String

Validation error detail array (optional)

10.2 Frontend Error Handling
	•	React Error Boundaries around major sections (Feed, Chat, Call) to prevent full-page crashes
	•	Axios/Fetch interceptors globally handle 401 responses by triggering silent token refresh
	•	Toast notifications (via shadcn/ui Toast) surface user-facing error messages
	•	Loading and error states managed per-query using TanStack Query (React Query)

13. Missing Details & Recommendations


The following items were not explicitly specified in the source documentation but are essential for a production-grade implementation:

13.1 Critical Gaps
Gap
Recommendation
Media storage service
Choose a cloud storage provider — AWS S3, Cloudflare R2, or Cloudinary. Without this, mediaUrls cannot be populated.
TURN server for WebRTC
Public STUN servers fail for ~15% of users behind strict NAT. Deploy Coturn on a VPS or use Twilio's TURN service for reliable production calls.
Content Delivery Network
Specify CDN provider for static assets and media. Vercel Edge Network covers frontend; a separate CDN is needed for user-uploaded media.
Database backup strategy
Define automated PostgreSQL backup schedule, retention policy, and point-in-time recovery approach.
Environment configuration
Document all required environment variables with descriptions, types, and example values in .env.example.
API rate limiting strategy
Define limits per role and endpoint (e.g. post creation: 20/hr for users; 100/hr for moderators).
Account deletion / GDPR
Implement user account deletion with cascade data removal or anonymisation to comply with data privacy regulations.
Accessibility (a11y)
Define WCAG 2.1 AA compliance targets. shadcn/ui provides a good baseline but custom components need auditing.
Monitoring & Alerting
Set up Uptime monitoring (e.g. Better Uptime) and alert thresholds for API error rates and latency.

13.2 Development Process Recommendations
	•	Adopt Conventional Commits for consistent git history and automated CHANGELOG generation
	•	Use Husky + lint-staged for pre-commit linting and formatting checks
	•	Define API versioning strategy (e.g. /api/v1/) from the start to allow future backwards-compatible changes
	•	Document all REST endpoints using OpenAPI/Swagger specification for team collaboration and client SDK generation
	•	Establish a branching strategy: main (production), develop (staging), feature/* branches with PR reviews
	•	Agree on a code review checklist covering security, performance, accessibility, and test coverage before merging

13.3 Scalability Considerations
	•	The Socket.IO server must use the Redis Pub/Sub adapter when deployed with multiple instances on Render
	•	Consider a message queue (BullMQ with Redis) for async tasks: notification sending, media processing, email delivery
	•	Implement database connection pooling (PgBouncer) to prevent connection exhaustion under load
	•	Design post feed queries with Keyset (cursor-based) pagination from day one — OFFSET queries degrade exponentially at scale
